# prd-maker 설계 문서

- 날짜: 2026-07-06
- 상태: 승인 대기 (브레인스토밍 완료, 사용자 설계 승인 완료)

## 1. 목적

아이디어를 가진 사람(비개발자 포함)이 적응형 인터뷰에 답하면, **어떤 AI 코딩 에이전트든 구현 지침으로 사용할 수 있는 범용 `PRD.md`** 를 생성해주는 Claude Code 플러그인.

핵심 문제 인식: AI 에이전트에게 일을 시킬 때 시작점인 PRD의 품질이 결과물의 품질을 결정한다. 그러나 좋은 에이전트용 PRD를 쓰는 방법(명시적 non-goals, 기계 검증 가능한 수용기준, 페이즈 분할 등)은 일반 사용자가 알기 어렵다. prd-maker는 이 지식을 인터뷰와 템플릿에 인코딩한다.

## 2. 확정된 요구사항

| 항목 | 결정 |
|---|---|
| 1차 사용자 | 비개발자 포함 누구나 |
| PRD 소비자 | 범용 — 어떤 AI 코딩 에이전트든 (특정 도구 문법 없는 순수 마크다운) |
| v1 범위 | 인터뷰 + PRD 생성만 (사전 리서치, 구현 핸드오프는 후속 버전) |
| 기술 결정 처리 | 사용자 기술 수준 감지 + 적응형 (개발자에겐 직접 질문, 비개발자에겐 스킬이 근거와 함께 기본값 도출) |
| 인터뷰 방식 | 적응형 + 커버리지 체크리스트 (자유 발화 먼저, 빈 요소만 한 번에 하나씩 질문) |
| 산출물 | 단일 `PRD.md`, 7섹션 구조 |
| PRD 언어 | 인터뷰에 사용된 사용자 언어를 따름 |
| 패키징 | Claude Code 플러그인 (`/prd-maker` 커맨드) |

## 3. 아키텍처

```
prd-maker/                          # 플러그인 루트 (이 저장소)
├── .claude-plugin/
│   └── plugin.json                 # 이름, 버전, 설명 메타데이터
├── commands/
│   └── prd-maker.md                # /prd-maker 커맨드 → 스킬 호출
└── skills/
    └── prd-maker/
        ├── SKILL.md                # 오케스트레이터 (150줄 이하)
        └── references/
            ├── interview-guide.md  # 어떻게 물을 것인가
            ├── prd-template.md     # 무엇을 산출할 것인가
            └── quality-rules.md    # 무엇이 좋은 PRD인가
```

### 컴포넌트 책임과 로드 시점

| 유닛 | 단일 책임 | 로드 시점 |
|---|---|---|
| `SKILL.md` | 4단계 워크플로우 지휘: ①아이디어 듣기 → ②적응형 인터뷰 → ③PRD 생성 → ④셀프 리뷰 후 전달. 엣지 케이스 규칙 포함 | 항상 |
| `interview-guide.md` | 기술 수준 감지법, 커버리지 체크리스트(8요소), 질문 전략, 비개발자용 질문 번역 | 인터뷰 시작 시 |
| `prd-template.md` | PRD 7섹션 구조 + 섹션별 작성 규칙 + 에이전트 지시 헤더 | PRD 작성 시 |
| `quality-rules.md` | 산출 직전 셀프 리뷰 체크리스트 | 셀프 리뷰 시 |

설계 원칙은 **점진적 공개(progressive disclosure)**: 각 단계에서 필요한 지침만 컨텍스트에 로드해 단계별 지시 개수를 낮게 유지한다. (근거: 프론티어 LLM도 동시 지시 150~200개 수준에서 수행 품질 저하 — 리서치 §7 참고)

### 데이터 플로우

사용자 아이디어 → 인터뷰로 수집한 답변들 → PRD 초안 → quality-rules 셀프 리뷰·수정 → 현재 디렉토리에 `PRD.md` 저장 → 사용자 검토 요청.

## 4. 인터뷰 플로우 (`interview-guide.md`)

**0단계 — 아이디어 듣기.** "만들고 싶은 것을 자유롭게 얘기해달라"고 요청. 형식 제약 없음.

**1단계 — 기술 수준 감지.** 별도 질문 없이 첫 발화에서 추정한다. 기술 용어("React로", "API 연동") → 개발자 트랙. 제품 수준 표현("앱 같은 거") → 비개발자 트랙. 애매하면 확인 질문 하나("평소 직접 코드를 작성하시나요?").

- **개발자 트랙**: 스택 선호, 기존 코드베이스 여부, 배포 환경을 직접 질문
- **비개발자 트랙**: 제품 수준 질문만 ("폰에서 쓰나요, 컴퓨터에서 쓰나요?", "혼자 쓰나요, 여러 명이 쓰나요?"). 기술 결정은 답변에서 스킬이 합리적 기본값을 도출해 **근거와 함께** PRD 기술 제약 섹션에 기록

**2단계 — 커버리지 체크리스트 기반 적응형 질문.** 8개 필수 요소 중 첫 발화에서 이미 채워진 것은 건너뛰고, 빈 것만 한 번에 하나씩 질문한다:

1. 해결하려는 문제 (왜)
2. 대상 사용자 (누구)
3. 핵심 기능 3~7개 (무엇)
4. Non-goals — 사용자가 언급하지 않으면 스킬이 후보를 제시해 확인 ("혹시 X는 이번 범위에서 빼는 게 맞나요?")
5. 성공 기준
6. 기술 제약/기존 결정 (트랙별 깊이 조절)
7. 우선순위 (페이즈 순서의 근거)
8. 데이터/콘텐츠 (다뤄야 할 정보)

**질문 규칙:**
- 선택지가 가능하면 객관식 우선
- "모르겠다"는 답에는 스킬이 추천안을 제시하고 동의 여부만 확인
- 총 질문 수 10개 이하 유지 — 우선순위 낮은 빈칸은 합리적 기본값으로 채우고 PRD에 `(가정)` 표시
- 이미 답한 내용은 다시 묻지 않는다

**종료 조건:** 8요소가 모두 채워지거나 기본값으로 대체되면, 수집 내용을 3~5줄로 요약해 사용자 확인 후 PRD 작성 단계로 진입.

## 5. PRD 템플릿 (`prd-template.md`)

산출물 `PRD.md`의 구조와 섹션별 작성 규칙:

**문서 머리 — 에이전트 지시 헤더:** "이 문서는 AI 에이전트가 구현 지침으로 사용한다. 불명확한 부분은 추측하지 말고 사용자에게 질문하라. 구현 중 결정이 바뀌면 이 문서를 갱신하라 (living document)."

1. **개요** — 문제와 목표 2~3문장. 규칙: 에이전트가 모호한 상황에서 판단 근거로 쓸 "왜"를 반드시 포함.
2. **대상 사용자 & JTBD** — 누가, 어떤 상황에서, 뭘 이루려고 쓰는가.
3. **핵심 기능 (스코프)** — 기능별 한 문단, 우선순위 순 정렬.
4. **Non-Goals** — 규칙: 생략이 아닌 양성 진술 ("이번 버전에서 사용자 인증은 구현하지 않는다"). 최소 3개.
5. **기술 제약 & 기존 결정** — 규칙: 비개발자 트랙에서 도출한 기본값은 근거 병기 ("혼자 쓰는 도구라 로그인 없이 로컬 저장 — 필요 시 에이전트가 변경 가능"). 재논의 불가 결정은 `[변경 금지]` 태그.
6. **페이즈별 요구사항** — 각 페이즈 = 목표 1문장 + 번호 매긴 요구사항 + 체크박스 수용기준. 규칙: 페이즈당 요구사항 30~50개 이하, 각 페이즈는 실행 가능한 상태로 종료(no dead ends), 페이즈 간 의존 순서 명시.
7. **성공 지표** — 측정 가능한 형태. 사용자가 정하지 않았으면 `(가정)` 표시.

범용 문서이므로 특정 도구(CLAUDE.md, .cursorrules 등) 문법은 사용하지 않는다.

## 6. 품질 룰 (`quality-rules.md`)

산출 직전 1회 셀프 리뷰 체크리스트 (위반 발견 시 해당 항목만 수정 후 산출, 재귀 리뷰 없음):

- 모든 수용기준이 기계 검증 가능한가? ("빠르게" ❌ → "3초 이내" ✅)
- Non-goals가 양성 진술인가? 스코프 크리프 위험 지점을 커버하는가?
- 페이즈 1만 구현해도 동작하는 결과물이 나오는가?
- 형용사·부사로만 된 요구사항("직관적인 UI")이 남아있는가?
- 사용자가 말하지 않은 내용을 지어내지 않았는가? 모든 가정에 `(가정)` 표시가 있는가?

## 7. 엣지 케이스 처리 (SKILL.md에 명시)

- **아이디어가 너무 큼** (여러 독립 서브시스템): 인터뷰 초반 감지 시 "v1은 가장 핵심 조각 하나"로 좁히기를 제안. 사용자가 전체를 고집하면 나머지는 Non-goals + "후속 버전" 항목으로 밀어냄.
- **무성의한/짧은 답변, 이탈 기미**: 다그치지 않고 남은 빈칸을 기본값 + `(가정)` 표시로 채우고 "가정이 많으니 검토해달라"고 명시.
- **`PRD.md` 이미 존재**: 덮어쓰기 전 확인 — 새 문서 생성 vs 기존 문서 읽고 갱신 중 선택하게 함.
- **기존 코드베이스 안에서 실행**: 코드가 있으면 스택을 가볍게 파악해 기술 제약 기본값으로 사용 (개발자 트랙이면 재확인).
- **답변 번복**: 최신 답변 우선, 조용히 갱신.

## 8. 검증 방법

프롬프트 자산이므로 시나리오 기반 수동 검증:

1. **비개발자 시나리오**: "동네 러닝 크루 모임 앱 만들고 싶어" → 기술 용어 질문이 나오지 않고, 기술 제약이 근거와 함께 자동 도출되는지 확인
2. **개발자 시나리오**: 스택 명시된 입력 → 이미 답한 체크리스트 항목을 재질문하지 않는지 확인
3. **에이전트 소비 스모크 테스트**: 산출된 PRD.md를 새 Claude Code 세션에 주고 "이 PRD대로 구현해줘" → 에이전트가 되묻지 않고 페이즈 1을 진행하면 합격. quality-rules를 평가 루브릭으로 재사용

각 시나리오의 세부 합격 기준은 구현 계획(writing-plans) 단계에서 태스크로 구체화한다.

## 9. 리서치 근거 요약

- 에이전트용 PRD는 "사람 간 합의 문서"가 아닌 **실행 가능한 인터페이스** — 스펙이 source of truth
- 6대 요소: outcomes, scope boundaries(명시적 non-goals), constraints, prior decisions, task breakdown, verification criteria
- 에이전트는 생략에서 경계를 추론하지 못함 → non-goals는 양성 진술 필수
- 동시 지시 150~200개 한계 → 페이즈 분할 + progressive disclosure
- 수용기준은 기계 검증 가능해야 함 ("API 응답 200ms 이하")
- Living document 원칙: 구현 중 결정 변경 시 스펙에 반영

## 10. v1 범위 밖 (후속 버전 후보)

- 작성 전 자동 리서치 페이즈 (유사 제품·기술 실현 가능성 웹서치)
- 구현 핸드오프 (페이즈별 태스크 분해, 구현 지시 프롬프트 생성)
- PRD 갱신 전용 모드 (living document 유지 보조)
- 다중 파일 산출 옵션 (PRD.md + phases/)
